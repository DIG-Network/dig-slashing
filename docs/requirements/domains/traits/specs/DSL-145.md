# DSL-145: InvalidBlockOracle::re_execute Determinism

> **Authoritative:** [DSL-145](../NORMATIVE.md#DSL-145) | **Verification:** [VERIFICATION.md](../VERIFICATION.md) | **Tracking:** [TRACKING.yaml](../TRACKING.yaml) | **SPEC:** §15.3

---

## Summary

`InvalidBlockOracle::re_execute(header, witness)` MUST be deterministic: identical `(header, witness)` inputs across any honest oracle implementation produce identical `ExecutionOutcome`. No I/O, no RNG, no hidden mutable state. Underpins invalid-block appeal grounds `BlockActuallyValid` (DSL-049) and `FailureReasonMismatch` (DSL-051).

---

## Specification

```rust
pub trait InvalidBlockOracle {
    fn re_execute(
        &self,
        header: &L2BlockHeader,
        witness: &[u8],
    ) -> Result<ExecutionOutcome, SlashingError>;

    fn verify_failure(
        &self,
        header: &L2BlockHeader,
        witness: &[u8],
        reason: InvalidBlockReason,
    ) -> Result<(), SlashingError> { Ok(()) }
}

pub enum ExecutionOutcome {
    Valid,
    Invalid(InvalidBlockReason),
}
```

---

## Acceptance Criteria

- [ ] Same `(header, witness)` → same `ExecutionOutcome` across calls.
- [ ] Different `header` OR different `witness` MAY produce different outcomes.
- [ ] `verify_failure` defaults to `Ok(())` for bootstrap path.
- [ ] No internal mutable state observable across calls.
