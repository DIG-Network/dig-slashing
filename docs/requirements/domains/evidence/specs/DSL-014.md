# DSL-014: verify_attester_slashing Double-Vote Predicate

> **Authoritative requirement:** [DSL-014](../NORMATIVE.md#DSL-014)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)
> **Spec reference:** [SPEC.md](../../../../resources/SPEC.md) — Sections 5.3, 22.2

---

## Summary

An `AttesterSlashing` is slashable under the DOUBLE-VOTE predicate when both attestations have the same `target.epoch` but different `AttestationData`. `verify_attester_slashing` accepts double-vote evidence, computes the intersection via `AttesterSlashing::slashable_indices`, and verifies BLS aggregate signatures on both sides.

---

## Specification

Double-vote predicate:
```text
a.data.target.epoch == b.data.target.epoch
AND a.data != b.data
```

Full verification:
1. `attestation_a.validate_structure()` AND `attestation_b.validate_structure()` (DSL-005).
2. `attestation_a != attestation_b` (byte-wise).
3. Double-vote OR surround-vote predicate holds (DSL-014 or DSL-015).
4. `attestation_a.verify_signature(...)` AND `attestation_b.verify_signature(...)` (DSL-006).
5. `slashable_indices = a ∩ b` non-empty (DSL-016).
6. Every slashable index is an active, non-slashed validator.

---

## Acceptance Criteria

- [ ] Two attestations with same target epoch + different data (shared committee) → Ok with intersection as slashable.
- [ ] Byte-identical attestations → `InvalidAttesterSlashing("identical")`.
- [ ] Different target epochs → falls through to surround-vote check (DSL-015), not accepted via DSL-014.
- [ ] Signatures verify on both sides (DSL-006).
- [ ] Intersection non-empty (DSL-016).
