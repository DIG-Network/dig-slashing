# DSL-020: verify_invalid_block Calls InvalidBlockOracle

> **Authoritative requirement:** [DSL-020](../NORMATIVE.md#DSL-020)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)
> **Spec reference:** [SPEC.md](../../../../resources/SPEC.md) — Sections 5.4, 15.3, 22.2

---

## Summary

When the caller supplies an `InvalidBlockOracle`, `verify_invalid_block` MUST call `oracle.verify_failure(&header, witness, reason)` and propagate any `Err`. Without an oracle, the default impl accepts any well-signed + well-formed witness (bootstrap path; admitted into the challenge window).

The oracle hook lets full-nodes re-execute the block and confirm invalidity before admitting evidence; light clients pass `None` and defer to the appeal-window challenge path.

---

## Specification

```rust
if let Some(oracle) = invalid_block_oracle {
    oracle.verify_failure(&payload.signed_header.message, &payload.failure_witness, payload.failure_reason)?;
}
```

---

## Acceptance Criteria

- [ ] Oracle = `Some(_)` → `verify_failure` called with `(header, witness, reason)`.
- [ ] Oracle returns `Err(_)` → error propagated.
- [ ] Oracle = `None` → skipped; evidence admitted under bootstrap semantics.
- [ ] Oracle's default impl returns `Ok(())`.

## Implementation Notes

- **Primary file:** `src/evidence/verify.rs` + trait in `src/traits.rs`.
- **Trait contract:** `InvalidBlockOracle::verify_failure` returns `Result<(), SlashingError>`.
