# DSL-127: run_epoch_boundary Fixed Ordering

> **Authoritative:** [DSL-127](../NORMATIVE.md#DSL-127) | **Verification:** [VERIFICATION.md](../VERIFICATION.md) | **Tracking:** [TRACKING.yaml](../TRACKING.yaml) | **SPEC:** ┬¦10, ┬¦22.15

## Summary

`run_epoch_boundary(...)` executes the epoch-boundary pipeline in a FIXED order. Reordering any step is a hard failure because downstream steps depend on earlier state. Returns an `EpochBoundaryReport` summarising flag deltas, inactivity penalties, finalisations, and stall state.

Order (normative):
1. Compute flag deltas over `participation.previous_epoch_all()`.
2. Update inactivity scores over the same previous-epoch flags.
3. Compute inactivity penalties for the ending epoch.
4. Finalise expired slashes (correlation penalty + reporter bond release + exit lock).
5. Rotate `ParticipationTracker` to `current_epoch_ending + 1`.
6. Advance `SlashingManager` epoch.
7. Resize trackers if `validator_count` changed.
8. Prune old processed evidence and correlation-window entries.


---

## Acceptance Criteria

- [ ] Step 1 runs before step 2; inactivity-score update cannot precede flag-delta computation.
- [ ] Step 3 runs before step 4 (inactivity penalties before finalisation).
- [ ] Step 4 (finalise) runs before step 5 (rotate); correlation window must see epoch before rotation.
- [ ] Step 8 (prune) is last.
- [ ] Report populated with `flag_deltas`, `inactivity_penalties`, `finalisations`, `in_finality_stall`.
- [ ] Reordering test (swap steps) fails.
