# DSL-053: InvalidBlockAppeal MissingOracle Error

> **Authoritative:** [DSL-053](../NORMATIVE.md#DSL-053) | **Verification:** [VERIFICATION.md](../VERIFICATION.md) | **Tracking:** [TRACKING.yaml](../TRACKING.yaml) | **SPEC:** §6.4, §15.3, §22.6

---

## Summary

Grounds `BlockActuallyValid` and `FailureReasonMismatch` require block re-execution via `InvalidBlockOracle`. When no oracle is supplied to `verify_appeal`, these grounds MUST return `AppealError::MissingOracle(name)` instead of a verdict.

Light clients typically lack re-execution capability; they defer adjudication to full nodes that supply an oracle.

---

## Specification

```rust
match ground {
    InvalidBlockAppealGround::BlockActuallyValid => {
        let oracle = invalid_block_oracle
            .ok_or(AppealError::MissingOracle("BlockActuallyValid requires InvalidBlockOracle"))?;
        ...
    }
    InvalidBlockAppealGround::FailureReasonMismatch => {
        let oracle = invalid_block_oracle
            .ok_or(AppealError::MissingOracle("FailureReasonMismatch requires InvalidBlockOracle"))?;
        ...
    }
    // Other grounds (ProposerSignatureInvalid, EvidenceEpochMismatch) do NOT require an oracle
    _ => { ... }
}
```

## Acceptance Criteria

- [ ] `FailureReasonMismatch` + no oracle → `MissingOracle`.
- [ ] `ProposerSignatureInvalid` works without oracle.
- [ ] `EvidenceEpochMismatch` works without oracle.
