# DSL-161: PendingSlash + AppealAttempt + PendingSlashStatus Serde Roundtrip

> **Authoritative:** [DSL-161](../NORMATIVE.md#DSL-161) | **Verification:** [VERIFICATION.md](../VERIFICATION.md) | **Tracking:** [TRACKING.yaml](../TRACKING.yaml) | **SPEC:** §3.8, §18

---

## Summary

`PendingSlash`, `PendingSlashStatus` (all 4 variants: Accepted, ChallengeOpen, Reverted, Finalised), and `AppealAttempt` MUST round-trip byte-exactly via bincode. Required for snapshot/restore and test fixture construction.

---

## Specification

```rust
#[derive(Debug, Clone, Copy, Serialize, Deserialize, PartialEq, Eq, Hash)]
pub enum PendingSlashStatus {
    Accepted,
    ChallengeOpen { first_appeal_filed_epoch: u64, appeal_count: u8 },
    Reverted  { winning_appeal_hash: Bytes32, reverted_at_epoch: u64 },
    Finalised { finalised_at_epoch: u64 },
}

#[derive(Debug, Clone, Serialize, Deserialize, PartialEq, Eq)]
pub struct AppealAttempt { /* ... */ }

#[derive(Debug, Clone, Serialize, Deserialize, PartialEq, Eq)]
pub struct PendingSlash { /* ... */ }
```

---

## Acceptance Criteria

- [ ] Each `PendingSlashStatus` variant round-trips.
- [ ] `AppealAttempt` with `Won` / `Lost { reason_hash }` / `Pending` round-trips.
- [ ] `PendingSlash` with non-empty `appeal_history` vec round-trips.
- [ ] Full nested tree (evidence + verified + status + per_validator + history) preserved.
