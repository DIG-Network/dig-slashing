# DSL-167: adjudicate_appeal Top-Level Dispatcher

> **Authoritative:** [DSL-167](../NORMATIVE.md#DSL-167) | **Verification:** [VERIFICATION.md](../VERIFICATION.md) | **Tracking:** [TRACKING.yaml](../TRACKING.yaml) | **SPEC:** §6.5

---

## Summary

Add a single public entry point `adjudicate_appeal(verdict, pending, appeal, validator_set, collateral, bond, reward, reward_clawback, current_epoch) -> AppealAdjudicationResult` that composes the 12 DSL-064..073 slice functions already shipped in `src/appeal/adjudicator.rs` into one end-to-end adjudication pass. Produces the `AppealAdjudicationResult` (DSL-164) type as its typed output.

Current state: the 12 slices (`adjudicate_sustained_revert_base_slash`, `adjudicate_sustained_restore_status`, `adjudicate_sustained_revert_collateral`, `adjudicate_sustained_forfeit_reporter_bond`, `adjudicate_sustained_reporter_penalty`, `adjudicate_absorb_clawback_shortfall`, `adjudicate_rejected_challenge_open`, `adjudicate_rejected_forfeit_appellant_bond`, `adjudicate_sustained_clawback_rewards`, `adjudicate_sustained_status_reverted`, plus the two inline status/history updates) are all `pub` but embedders must hand-compose them. This requirement closes that gap.

---

## Specification

```rust
pub fn adjudicate_appeal(
    verdict: AppealVerdict,
    pending: &mut PendingSlash,
    appeal: &SlashAppeal,
    validator_set: &mut dyn ValidatorView,
    collateral: &mut dyn CollateralSlasher,
    bond_escrow: &mut dyn BondEscrow,
    reward_payout: &mut dyn RewardPayout,
    reward_clawback: &mut dyn RewardClawback,
    current_epoch: u64,
) -> AppealAdjudicationResult;
```

### Sustained branch (fixed order)

1. `adjudicate_sustained_revert_base_slash` → `reverted_stake_mojos`.
2. `adjudicate_sustained_revert_collateral` → `reverted_collateral_mojos`.
3. `adjudicate_sustained_restore_status`.
4. `adjudicate_sustained_clawback_rewards` → `clawback_shortfall`.
5. `adjudicate_sustained_forfeit_reporter_bond` → `reporter_bond_forfeited`, `appellant_award_mojos`, partial `burn_amount`.
6. `adjudicate_absorb_clawback_shortfall` (residue adjustment; may reduce `appellant_award`, may increase `burn_amount`).
7. `adjudicate_sustained_reporter_penalty` → `reporter_penalty_mojos`.
8. `adjudicate_sustained_status_reverted` + append `AppealAttempt { outcome: Won }` to `pending.appeal_history`.

### Rejected branch (fixed order)

1. `adjudicate_rejected_forfeit_appellant_bond` → `appellant_bond_forfeited`, `reporter_award_mojos`, `burn_amount`.
2. `adjudicate_rejected_challenge_open` → `pending.status = ChallengeOpen { first_appeal_filed_epoch: min(existing, filed), appeal_count: n + 1 }`, append `AppealAttempt { outcome: Lost { reason_hash } }` to `pending.appeal_history`.

### Outcome conversion

`result.outcome = verdict.to_appeal_outcome()` (DSL-171). Sustained → `Won`; Rejected{reason} → `Lost { reason_hash }`.

---

## Acceptance Criteria

- [ ] Sustained path produces populated `reverted_stake_mojos` + `reverted_collateral_mojos` + awards; rejected-branch fields stay zero.
- [ ] Rejected path produces `reporter_award_mojos` + `appellant_bond_forfeited`; sustained-branch fields stay empty/zero.
- [ ] `pending.status` post-Sustained is `Reverted { winning_appeal_hash = appeal.hash(), reverted_at_epoch = current_epoch }`.
- [ ] `pending.status` post-Rejected is `ChallengeOpen { first_appeal_filed_epoch, appeal_count }` with `appeal_count` incremented.
- [ ] `pending.appeal_history` grows by exactly one entry per call.
- [ ] Sustained slice order matches the numbered list above (clawback BEFORE forfeit BEFORE shortfall absorption).
- [ ] Result `outcome` matches `verdict.to_appeal_outcome()`.
- [ ] No panic when `collateral` is a no-op impl (DSL-139 default) or `reward_clawback` returns partial (DSL-142).
