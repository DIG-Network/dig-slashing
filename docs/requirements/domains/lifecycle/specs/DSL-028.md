# DSL-028: submit_evidence Rejects BondLockFailed on Insufficient Stake

> **Authoritative requirement:** [DSL-028](../NORMATIVE.md#DSL-028)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)
> **Spec reference:** [SPEC.md](../../../../resources/SPEC.md) — Sections 7.3 (step 4), 12.3, 17, 22.3

---

## Summary

A reporter MUST have enough unlocked stake to cover `REPORTER_BOND_MOJOS` at the moment of admission. Otherwise the bond escrow returns `BondError::InsufficientBalance` and `SlashingManager::submit_evidence` MUST convert this into `SlashingError::BondLockFailed` and abort the admission. No validator state MUST be mutated, no rewards MUST be paid, and no `PendingSlash` MUST be inserted on this path.

This ensures the bond is always fully collateralised by real stake: if the reporter cannot post it, the slash cannot be admitted. Without this constraint, an adversary could flood the network with evidence they cannot back financially. The reporter's own stake is the safety deposit against frivolous reports.

The check happens inside `BondEscrow::lock` (implementation lives in the `bonds/` domain, DSL-121). The manager's role is to (a) call `lock` at the correct point in the pipeline and (b) translate the error.

---

## Specification

### Error Mapping

```rust
bond_escrow
    .lock(reporter_idx, REPORTER_BOND_MOJOS, BondTag::Reporter(hash))
    .map_err(|e| match e {
        BondError::InsufficientBalance => SlashingError::BondLockFailed,
        other => SlashingError::BondOther(other),
    })?;
```

### Ordering

The bond lock is attempted **after** the capacity check (DSL-027) and **before** any validator mutation (DSL-022) or reward payout (DSL-025). Therefore:

- On `BondLockFailed`, no validator has been debited.
- No reward has been paid.
- No `PendingSlash` has been inserted.
- `processed` has NOT been updated (re-submission with a funded reporter is allowed later).

### Invariants

| Invariant | Description |
|-----------|-------------|
| Pure rejection | All external state (`validator_set`, `reward_payout`, `book`, `processed`) is byte-identical before and after |
| Re-attempt allowed | Since `processed` is not updated, the reporter MAY try again after posting more stake |
| Error variant | MUST be `SlashingError::BondLockFailed` (not `AlreadySlashed`, not `PendingBookFull`) |

## Acceptance Criteria

- [ ] On `BondError::InsufficientBalance`, returns `Err(SlashingError::BondLockFailed)`.
- [ ] No `slash_absolute` call recorded by the validator set mock.
- [ ] No `pay` call recorded by the reward payout mock.
- [ ] `book.len()` unchanged.
- [ ] `processed.contains_key(hash)` is `false` after rejection.
- [ ] `BondEscrow::escrowed(reporter_idx, tag) == 0` after rejection.
- [ ] Re-submitting the same evidence after funding the reporter succeeds.

## Implementation Notes

- **Primary file:** `src/manager.rs`.
- **Reuses:** `BondEscrow` trait (`bonds/` domain, DSL-121..126).
- **Edge cases:**
  - Reporter has been slashed in this very submission: because `reporter_is_accused` is rejected earlier in `verify_evidence` (SPEC §5.1 precondition 3), this path is structurally impossible — the reporter is never among the accused.
  - Reporter's stake dropped below bond amount due to a prior slash in the same epoch: this is the intended failure mode; the report is rejected.
  - Zero-value bond (degenerate config): `lock(_, 0, _)` is a no-op and cannot fail with `InsufficientBalance`.
