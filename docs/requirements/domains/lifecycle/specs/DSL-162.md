# DSL-162: submit_evidence Skips Already-Slashed Indices

> **Authoritative:** [DSL-162](../NORMATIVE.md#DSL-162) | **Verification:** [VERIFICATION.md](../VERIFICATION.md) | **Tracking:** [TRACKING.yaml](../TRACKING.yaml) | **SPEC:** §7.3

---

## Summary

Within `submit_evidence`'s per-validator slash loop (DSL-022 step 5), every index returned by `VerifiedEvidence::slashable_validator_indices` MUST be checked with `ValidatorEntry::is_slashed()`. Indices already in `Slashed` state MUST be skipped: NO stake debit, NO collateral debit, NO `slashed_in_window` record, NO `PerValidatorSlash` entry in the result vec.

Defensive: covers the gap between `verify_evidence` (which only checks slashed-status for proposer evidence) and `AttesterSlashing` cases where some indices in the intersection may have been slashed by a prior admission.

---

## Specification

for idx in verified.slashable_validator_indices {

    if entry.is_slashed() { continue; }  // DSL-162: uniform skip

---

## Acceptance Criteria

- [ ] Index flagged `is_slashed()` → skipped uniformly.
- [ ] Result `SlashingResult::per_validator` omits skipped indices.
- [ ] No `slash_absolute` call on skipped indices.
- [ ] No `slashed_in_window` entry inserted for skipped.
- [ ] AttesterSlashing case: if 3 of 5 intersection indices are already slashed, only 2 slashes are applied.
- [ ] If ALL indices already slashed, result vec is empty but evidence is still marked `processed` (dedup).
