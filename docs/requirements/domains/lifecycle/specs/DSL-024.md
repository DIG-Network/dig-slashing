# DSL-024: submit_evidence Creates PendingSlash in Accepted Status

> **Authoritative requirement:** [DSL-024](../NORMATIVE.md#DSL-024)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)
> **Spec reference:** [SPEC.md](../../../../resources/SPEC.md) — Sections 3.8, 7.1, 7.3 (steps 12–13), 22.3

---

## Summary

Optimistic slashing is reversible, so the manager keeps a book of **pending slashes** whose fate is decided during the appeal window. `SlashingManager::submit_evidence`, after verification and all economic effects (bond lock, validator slashes, reward routing), MUST insert a `PendingSlash` record into `PendingSlashBook` with:

- `status = PendingSlashStatus::Accepted`
- `submitted_at_epoch = current_epoch`
- `window_expires_at_epoch = current_epoch + SLASH_APPEAL_WINDOW_EPOCHS` (i.e. `current_epoch + 8`)
- `base_slash_per_validator` containing one `PerValidatorSlash` entry per slashed validator
- `reporter_bond_mojos = REPORTER_BOND_MOJOS`
- `appeal_history = Vec::new()` (no attempts yet)

The `evidence_hash` MUST simultaneously be registered in the manager's `processed: HashMap<Bytes32, u64>` map with value `current_epoch`. This ensures duplicate submissions (DSL-026) are rejected immediately, and the book's by-window-expiry index (`BTreeMap<u64, Vec<Bytes32>>`) is maintained for efficient `expired_by(epoch)` scans at finalisation (DSL-029).

The `Accepted` status is the starting state; subsequent transitions are `ChallengeOpen` (on first appeal), `Reverted` (on sustained appeal), or `Finalised` (on window expiry without a sustained appeal).

## Specification

### Data Model

```rust
// SPEC §3.8
pub enum PendingSlashStatus {
    Accepted,
    ChallengeOpen { first_appeal_filed_epoch: u64, appeal_count: u8 },
    Reverted  { winning_appeal_hash: Bytes32, reverted_at_epoch: u64 },
    Finalised { finalised_at_epoch: u64 },
}

pub struct PendingSlash {
    pub evidence_hash: Bytes32,
    pub evidence: SlashingEvidence,
    pub verified: VerifiedEvidence,
    pub status: PendingSlashStatus,
    pub submitted_at_epoch: u64,
    pub window_expires_at_epoch: u64,
    pub base_slash_per_validator: Vec<PerValidatorSlash>,
    pub reporter_bond_mojos: u64,
    pub appeal_history: Vec<AppealAttempt>,
}
```

### Insertion

```rust
// Inside submit_evidence, after all economic side-effects:
let pending = PendingSlash {
    evidence_hash: hash,
    evidence,
    verified,
    status: PendingSlashStatus::Accepted,
    submitted_at_epoch: self.current_epoch,
    window_expires_at_epoch: self.current_epoch + SLASH_APPEAL_WINDOW_EPOCHS,
    base_slash_per_validator,
    reporter_bond_mojos: REPORTER_BOND_MOJOS,
    appeal_history: Vec::new(),
};
self.book.insert(pending)?;
self.processed.insert(hash, self.current_epoch);
```

### Invariants

| Invariant | Description |
|-----------|-------------|
| Status on insert | MUST be `Accepted`; never `ChallengeOpen/Reverted/Finalised` on first insert |
| Window width | `window_expires_at_epoch - submitted_at_epoch == SLASH_APPEAL_WINDOW_EPOCHS` |
| Per-validator count | `base_slash_per_validator.len() == verified.slashable_validator_indices.len() - skipped` |
| `processed` entry | `processed[hash] == current_epoch` |
| `by_window_expiry` index | `hash` is appended to the vec at key `window_expires_at_epoch` |

## Acceptance Criteria

- [ ] `PendingSlashBook::insert` is called exactly once per successful submit.
- [ ] Inserted record has `status == PendingSlashStatus::Accepted`.
- [ ] `submitted_at_epoch == self.current_epoch`.
- [ ] `window_expires_at_epoch == self.current_epoch + SLASH_APPEAL_WINDOW_EPOCHS`.
- [ ] `base_slash_per_validator` populated with one entry per actually-slashed validator.
- [ ] `reporter_bond_mojos == REPORTER_BOND_MOJOS`.
- [ ] `appeal_history.is_empty()` on admission.
- [ ] `processed.insert(hash, current_epoch)` is called after successful book insert.
- [ ] `SlashingResult::pending_slash_hash` equals `evidence.hash()`.

---

## Implementation Notes

- **Primary file:** `src/manager.rs` + `src/pending.rs` (`PendingSlashBook`).
- **Reuses:** `PendingSlashBook::insert` which maintains both `pending` and `by_window_expiry` secondary index.
- **Edge cases:**
  - Book at capacity: DSL-027 covers `PendingBookFull` rejection; this requirement targets only the successful path.
  - If no validators were actually slashed (all skipped as already-slashed), `base_slash_per_validator` is empty — but `verify_evidence` would have returned `EmptySlashableIntersection` earlier.
  - Insertion ordering MUST be atomic with `processed`: rollback `processed` insert if `book.insert` fails (though with DSL-027 check up front the failure path is narrow).
