# DSL-027: submit_evidence Rejects PendingBookFull at Capacity

> **Authoritative requirement:** [DSL-027](../NORMATIVE.md#DSL-027)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)
> **Spec reference:** [SPEC.md](../../../../resources/SPEC.md) — Sections 2.6, 7.1, 7.3 (step 3), 17, 22.3

---

## Summary

The `PendingSlashBook` has a bounded capacity of `MAX_PENDING_SLASHES = 4_096` to prevent unbounded memory growth in the presence of adversarial or accidental slash-submission storms. `SlashingManager::submit_evidence` MUST check `book.len() >= MAX_PENDING_SLASHES` before attempting to lock the reporter bond or mutate validator state, and MUST return `Err(SlashingError::PendingBookFull)` if the check fails.

The check is placed after dedup (DSL-026) and verification (so only valid, non-duplicate evidence can trigger capacity exhaustion) but **before** `BondEscrow::lock` and before any `slash_absolute` calls. This ordering ensures no reporter bond is locked for a slash that cannot be inserted, and no validator is debited for a slash that will be rolled back.

Capacity is measured against the `pending: HashMap` cardinality (not the `by_window_expiry` index). Items move out of the book only when `rewind_on_reorg` drops them or when a future pruning pass removes Finalised/Reverted entries older than their window.

## Specification

### Check Location

```rust
pub fn submit_evidence(&mut self, evidence: SlashingEvidence, ...)
    -> Result<SlashingResult, SlashingError>
{
    let hash = evidence.hash();
    if self.processed.contains_key(&hash) {
        return Err(SlashingError::AlreadySlashed);
    }

    let verified = verify_evidence(&evidence, validator_set, network_id, self.current_epoch)?;

    // Step 3 — capacity check; BEFORE bond lock
    if self.book.len() >= MAX_PENDING_SLASHES {
        return Err(SlashingError::PendingBookFull);
    }

    bond_escrow.lock(reporter_idx, REPORTER_BOND_MOJOS, BondTag::Reporter(hash))
        .map_err(|_| SlashingError::BondLockFailed)?;
    // ...
}
```

### Constants in scope

| Constant | Value | Source |
|----------|-------|--------|
| `MAX_PENDING_SLASHES` | `4_096` | SPEC §2.6 |

### Invariants

| Invariant | Description |
|-----------|-------------|
| No bond lock | `BondEscrow::lock` is NOT called on this error path |
| No validator mutation | `ValidatorEntry::slash_absolute` is NOT called |
| No reward payout | `RewardPayout::pay` is NOT called |
| `processed` unchanged | Hash remains absent from `processed`; re-submission can succeed once capacity frees |
| Strict `>=` | Check fails when `book.len() == capacity`; the book never holds `capacity + 1` items |

---

## Acceptance Criteria

- [ ] Capacity check appears after verification and before bond lock.
- [ ] Returns `Err(SlashingError::PendingBookFull)` when `book.len() >= MAX_PENDING_SLASHES`.
- [ ] `BondEscrow::lock`, `slash_absolute`, and `pay` are NOT called on rejection.
- [ ] `book.len()` is unchanged after rejection.
- [ ] `processed.contains_key(hash)` is `false` after rejection — allows retry.
- [ ] Threshold is exclusive: at `len == capacity - 1` the submit still succeeds.

---

## Implementation Notes

- **Primary file:** `src/manager.rs`; `src/pending.rs` exposes `PendingSlashBook::len`.
- **Reuses:** `PendingSlashBook::len` and the `MAX_PENDING_SLASHES` constant.
- **Edge cases:**
  - Concurrent submits: the manager is single-threaded by contract; no TOCTOU concern.
  - Reorgs: `rewind_on_reorg` drops pending slashes, freeing capacity naturally.
  - Book currently has no in-place pruning for Finalised entries; capacity is a true hard ceiling until retention logic is added.
