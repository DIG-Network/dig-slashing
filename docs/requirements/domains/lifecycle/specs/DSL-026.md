# DSL-026: submit_evidence Rejects AlreadySlashed on Duplicate

> **Authoritative requirement:** [DSL-026](../NORMATIVE.md#DSL-026)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)
> **Spec reference:** [SPEC.md](../../../../resources/SPEC.md) — Sections 7.2, 7.3 (step 1), 17 (SlashingError), 22.3

---

## Summary

`SlashingManager::submit_evidence` MUST reject a re-submission of evidence that has already been processed (even if its pending slash is now `Reverted` or `Finalised`). The manager maintains a `processed: HashMap<Bytes32, u64>` map keyed by `evidence.hash()` whose values are the admission epoch. The very first step of the pipeline (before verification, before the book-capacity check, before any bond lock) looks up `evidence.hash()` in `processed`; if present, the manager returns `Err(SlashingError::AlreadySlashed)`.

This protects against three scenarios:

1. **Honest re-admission** — the same evidence was relayed twice by different gossip peers.
2. **Replay attack** — an adversary re-submits an old, already-adjudicated slash.
3. **Reorg-adjacent replay** — during a reorg, stale evidence could hit the mempool again; rewind-on-reorg must explicitly clear `processed` for rolled-back epochs (see DSL-129).

Critically, **no state is mutated** on this path: no bond is locked, no validator is slashed, no reward is paid, and no `PendingSlash` is inserted. Dedup MUST be the first check in the pipeline (SPEC §7.3 step 1).

## Specification

### Check Location

```rust
pub fn submit_evidence(&mut self, evidence: SlashingEvidence, ...)
    -> Result<SlashingResult, SlashingError>
{
    let hash = evidence.hash();

    // Step 1 — dedup (earliest possible; no state mutation above this line)
    if self.processed.contains_key(&hash) {
        return Err(SlashingError::AlreadySlashed);
    }

    // Step 2 onwards: verify_evidence, capacity check, bond lock, slashes, ...
}
```

### Error

`SlashingError::AlreadySlashed` — shared with `verify_evidence` which rejects the same condition when a validator is already flagged `is_slashed()` (SPEC §5.2 step 5). In the manager context, "already slashed" refers to the evidence-hash-level duplicate.

### Invariants

| Invariant | Description |
|-----------|-------------|
| No state mutation | `validator_set`, `bond_escrow`, `reward_payout`, `book`, `processed` all unchanged |
| Earliest check | Dedup precedes `verify_evidence` and `book.is_full()` — cheapest rejection path |
| Persistence across statuses | `processed` entry remains even after pending slash becomes `Reverted` or `Finalised` |
| `is_processed` helper | `SlashingManager::is_processed(hash)` returns `true` iff hash is in `processed` |

---

## Acceptance Criteria

- [ ] Dedup check is the first statement after `hash = evidence.hash()`.
- [ ] Returns `Err(SlashingError::AlreadySlashed)` on duplicate.
- [ ] No call to `verify_evidence`, `bond_escrow.lock`, `validator_set.get_mut`, `reward_payout.pay`, or `book.insert` on duplicate path.
- [ ] Duplicate check holds across pending statuses (Accepted, ChallengeOpen, Reverted, Finalised).
- [ ] `SlashingManager::is_processed(hash)` returns `true` after first successful submit.

---

## Implementation Notes

- **Primary file:** `src/manager.rs`.
- **Reuses:** `SlashingEvidence::hash()` (DSL-002) — deterministic 32-byte digest.
- **Edge cases:**
  - Reorg: `rewind_on_reorg` (DSL-129) MUST remove rolled-back hashes from `processed` so that re-admission in the new branch is allowed.
  - Prune: `SlashingManager::prune(before_epoch)` MAY remove old `processed` entries, but only for evidence whose pending slash has already been `Finalised` and whose `window_expires_at_epoch < before_epoch`.
