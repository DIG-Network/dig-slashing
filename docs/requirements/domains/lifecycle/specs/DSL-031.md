# DSL-031: finalise_expired_slashes Returns Reporter Bond in Full

> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)
> **Sec reference:** [SPEC.md](../../../../resources/SPEC.md) — Sections 2.6, 7.4 (step 5), 12.3, 22.3

## Summary

When a pending slash finalises without a sustained appeal, the reporter's bond has served its purpose — no fraudulent evidence was exposed, so no penalty applies. `SlashingManager::finalise_expired_slashes` MUST return the bond in full by calling `BondEscrow::release(reporter_idx, REPORTER_BOND_MOJOS, BondTag::Reporter(evidence_hash))`.

"In full" means the exact amount originally locked (DSL-023) returns to the reporter's spendable stake. Unlike forfeiture (which applies on sustained appeal per DSL-068), release preserves the bond balance entirely. The escrow implementation simply zeroes the tagged slot and credits the reporter's stake.

The release is recorded in `FinalisationResult::reporter_bond_returned`. This field MUST equal `REPORTER_BOND_MOJOS` on every successful finalisation.

The release fires for both `Accepted → Finalised` and `ChallengeOpen → Finalised` transitions (DSL-029), since in both cases the slash stands and no sustained appeal was filed. The bond is NOT released for `Reverted → <skip>` entries (DSL-033) — those entries had their bond forfeited on the sustained appeal (DSL-068).

## Specification

### Call Site

```rust
// Inside finalise_expired_slashes per-pending loop, after correlation penalty:
bond_escrow
    .release(
        pending.evidence.reporter_validator_index,
        REPORTER_BOND_MOJOS,
        BondTag::Reporter(pending.evidence_hash),
    )
    .expect("release of previously-locked bond is infallible by construction");

result.reporter_bond_returned = REPORTER_BOND_MOJOS;
```

### Invariants

| Invariant | Description |
|-----------|-------------|
| Exact-amount release | `release(reporter_idx, REPORTER_BOND_MOJOS, Reporter(hash))` — no partial release |
| Infallible by construction | The bond was locked at admission; escrow accounting guarantees release cannot fail for a valid tag |
| Result field | `FinalisationResult::reporter_bond_returned == REPORTER_BOND_MOJOS` |
| No double-release | Idempotent finalise (DSL-029) prevents calling `release` twice for the same tag |
| Skip Reverted | Bond release does NOT occur for `Reverted` pendings (see DSL-033) — those were forfeited |

---

## Acceptance Criteria

- [ ] `BondEscrow::release` is called exactly once per finalised pending slash.
- [ ] Arguments: `(reporter_idx, REPORTER_BOND_MOJOS, BondTag::Reporter(evidence_hash))`.
- [ ] `FinalisationResult::reporter_bond_returned == REPORTER_BOND_MOJOS` on every finalised entry.
- [ ] Not called for pendings transitioning from `Reverted` (they are skipped entirely per DSL-033).
- [ ] Release occurs after correlation penalty (DSL-030) and before/alongside exit-lock scheduling (DSL-032).
- [ ] Post-release, `BondEscrow::escrowed(reporter_idx, Reporter(hash)) == 0`.

---

## Implementation Notes

- **Primary file:** `src/manager.rs`.
- **Reuses:** `BondEscrow::release` from `bonds/` domain (DSL-123).
- **Edge cases:**
  - Escrow implementation returning an unexpected error here indicates a book-keeping bug; implementations MAY `expect` on the result since a lock that succeeded at admission should always release cleanly. Alternatively, log + continue to avoid blocking epoch advancement.
  - Reporter has exited the validator set between admission and finalisation: the escrow tracks mojos by `principal_idx`, so the bond returns to the slot regardless of active status.
