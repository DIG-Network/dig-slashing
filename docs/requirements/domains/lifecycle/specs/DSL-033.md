# DSL-033: finalise_expired_slashes Skips Already-Reverted Slashes

> **Authoritative requirement:** [DSL-033](../NORMATIVE.md#DSL-033)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)
> **Spec reference:** [SPEC.md](../../../../resources/SPEC.md) — Sections 3.8, 7.4 (step 2), 22.3

---

## Summary

A pending slash can reach the expiry epoch in any of four statuses:

1. **Accepted** — no appeals; finalise normally (DSL-029..032).
2. **ChallengeOpen** — appeals filed but all rejected; finalise normally.
3. **Reverted** — a sustained appeal already reversed the slash (the adjudicator applied the reversal side-effects and this pending is effectively dead-letter).
4. **Finalised** — already finalised in a prior epoch-boundary call (idempotency).

For statuses (3) and (4), `finalise_expired_slashes` MUST be a no-op: **no validator mutation**, **no bond release**, **no exit-lock scheduling**, **no FinalisationResult emission**. The pending stays in the book as a historical record until pruning.

This is the defensive guard at SPEC §7.4 step 2: `If pending.status == Reverted or Finalised → skip`. It protects correctness against two scenarios:

- **Reverted**: the sustained appeal path (DSL-068..070) already credited back the base slash, clawed back rewards, and forfeited the reporter bond. Re-running correlation penalty + bond release would double-count or double-apply — catastrophic.
- **Finalised**: a previous epoch's call already handled this entry. Without the guard, `expired_by` would return it again (it still matches the `<= current_epoch` predicate) and the system would double-finalise.

## Specification

### Check Location

```rust
for hash in self.book.expired_by(self.current_epoch) {
    let pending = match self.book.get_mut(&hash) {
        Some(p) => p,
        None    => continue,
    };

    // Step 2 — defensive skip
    match pending.status {
        PendingSlashStatus::Reverted  { .. } => continue,
        PendingSlashStatus::Finalised { .. } => continue,
        PendingSlashStatus::Accepted
        | PendingSlashStatus::ChallengeOpen { .. } => {
            // continue to correlation penalty, bond release, exit lock, transition
        }
    }
    // ...
}
```

### Invariants

| Invariant | Description |
|-----------|-------------|
| No validator mutation | `ValidatorEntry::slash_absolute` not called on skipped entries |
| No exit-lock | `schedule_exit` not called on skipped entries |
| No bond release | `BondEscrow::release` not called on skipped entries (Reverted already forfeited it; Finalised already released) |
| No result emission | `FinalisationResult` NOT appended to return vec for skipped entries |
| Pending retained | The skipped `PendingSlash` remains in `book.pending` and in `book.by_window_expiry` |
| Idempotency | Calling `finalise_expired_slashes` repeatedly has no cumulative effect |

---

## Acceptance Criteria

- [ ] Pendings with `PendingSlashStatus::Reverted { .. }` are skipped entirely.
- [ ] Pendings with `PendingSlashStatus::Finalised { .. }` are skipped entirely (idempotency).
- [ ] No validator, bond, or reward-related trait method is called for skipped entries.
- [ ] Skipped entries do NOT produce a `FinalisationResult`.
- [ ] The pending record remains in the book after the skip.
- [ ] A subsequent `finalise_expired_slashes` call in the same epoch produces an empty result vec.

---

## Implementation Notes

- **Primary file:** `src/manager.rs`.
- **Reuses:** `PendingSlashStatus` (SPEC §3.8); `PendingSlashBook` (SPEC §7.1).
- **Edge cases:**
  - A pending that was `Reverted` between two consecutive finalise calls: first call sees it as Reverted (because the sustained appeal ran); second call skips again.
  - `Accepted` and `ChallengeOpen` pendings process normally — the skip pattern is an exhaustive match so any future status variants force a compile-time decision.
