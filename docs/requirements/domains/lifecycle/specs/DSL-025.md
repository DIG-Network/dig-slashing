## Summary

Successful evidence submission pays out two optimistic rewards immediately (before the appeal window elapses):

1. **Whistleblower reward** (`wb_reward`) â€” paid to the reporter's puzzle hash. Size: `total_eff_bal / WHISTLEBLOWER_REWARD_QUOTIENT` = `total_eff_bal / 512`.
2. **Proposer reward** (`prop_reward`) â€” paid to the block-proposer-at-current-slot's puzzle hash (the node that *includes* the evidence in a block). Size: `wb_reward / PROPOSER_REWARD_QUOTIENT` = `wb_reward / 8`.

Both payments are routed through the `RewardPayout` trait (Â§12.1), which in the consensus layer creates or credits a pay-to-puzzle-hash reward coin. `total_eff_bal` is the sum of `EffectiveBalanceView::get(idx)` across every validator that was actually slashed.

The remaining slash mojos are burned: `burn_amount = total_base_slash - wb_reward - prop_reward`.

Both rewards are **reversible**: if a sustained appeal wins during the window, `RewardClawback::claw_back` pulls these mojos back (see appeal domain DSL-067).

The block-proposer puzzle hash is obtained through two dependencies:

- `ProposerView::proposer_at_slot(proposer.current_slot())` returns the validator index.
- `ValidatorView::get(idx).puzzle_hash()` returns the payout puzzle hash.

## Specification

### Computation

```rust
// After the per-validator loop has populated per_validator and totals:
let total_eff_bal: u64 = per_validator.iter().map(|p| p.effective_balance_at_slash).sum();
let total_base:  u64  = per_validator.iter().map(|p| p.base_slash_amount).sum();

let wb_reward   = total_eff_bal / WHISTLEBLOWER_REWARD_QUOTIENT; // 512
let prop_reward = wb_reward     / PROPOSER_REWARD_QUOTIENT;      // 8
let burn_amount = total_base.saturating_sub(wb_reward + prop_reward);
```

### Payout Calls

```rust
// Reporter whistleblower reward
reward_payout.pay(evidence.reporter_puzzle_hash, wb_reward);

// Block-proposer inclusion reward
let proposer_idx = proposer
    .proposer_at_slot(proposer.current_slot())
    .ok_or(SlashingError::ProposerUnavailable)?;
let proposer_ph  = validator_set.get(proposer_idx).unwrap().puzzle_hash();
reward_payout.pay(proposer_ph, prop_reward);
```

## Acceptance Criteria

- [ ] `RewardPayout::pay` is called exactly twice per successful submit â€” once with reporter puzzle hash, once with proposer puzzle hash.
- [ ] Reporter payment amount equals `total_eff_bal / 512`.
- [ ] Proposer payment amount equals `wb_reward / 8`.
- [ ] `SlashingResult::whistleblower_reward` and `::proposer_reward` match the paid amounts.
- [ ] `SlashingResult::burn_amount == total_base_slash - wb_reward - prop_reward`.
- [ ] Block-proposer puzzle hash derived via `ProposerView::proposer_at_slot(current_slot)` then `ValidatorView::get(idx).puzzle_hash()`.
- [ ] Rewards paid AFTER all `slash_absolute` calls and BEFORE `book.insert` (ordering preserves atomicity of `SlashingResult`).

## Implementation Notes

- **Primary file:** `src/manager.rs`.
- **Reuses:** `RewardPayout` trait (Â§12.1); `ProposerView` trait (Â§15.3); `ValidatorEntry::puzzle_hash` (Â§15.1).
- **Edge cases:**
  - `total_eff_bal < 512` â†’ `wb_reward == 0` â†’ `prop_reward == 0` â†’ `burn_amount == total_base`. The two `pay(..., 0)` calls MUST still occur (keeps call pattern deterministic for auditors).
  - `ProposerView::proposer_at_slot` returning `None` at admission is a consensus-layer bug; the manager surfaces a `ProposerUnavailable` error rather than silently dropping the reward.
  - Integer division is exact; no rounding policy needed.
