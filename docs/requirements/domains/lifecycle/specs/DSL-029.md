# DSL-029: finalise_expired_slashes Transitions Accepted/ChallengeOpen to Finalised

> **Authoritative requirement:** [DSL-029](../NORMATIVE.md#DSL-029)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)
> **Spec reference:** [SPEC.md](../../../../resources/SPEC.md) — Sections 3.8, 7.1, 7.4 (steps 1, 6–7), 22.3

## Summary

At every epoch boundary (before new-epoch block production), the consensus layer calls `SlashingManager::finalise_expired_slashes`. The method iterates over every `evidence_hash` returned by `PendingSlashBook::expired_by(current_epoch)` — i.e. every pending slash whose `window_expires_at_epoch <= current_epoch` — and:

- If the pending slash status is `Accepted` (no appeals filed) or `ChallengeOpen { .. }` (appeals filed but none sustained), the status MUST transition to `PendingSlashStatus::Finalised { finalised_at_epoch: current_epoch }`.
- A `FinalisationResult` MUST be emitted and returned in the caller's `Vec<FinalisationResult>`.

The `Accepted` → `Finalised` transition is the happy path: no one challenged the slash, the optimistic debit becomes permanent, correlation penalty fires (DSL-030), reporter bond is refunded (DSL-031), and the exit lock starts (DSL-032). The `ChallengeOpen` → `Finalised` transition fires when appeals were filed but all were rejected; the slash persists unchanged in terms of base debit.

This requirement targets only the **status transition and result emission**. The actual penalty, bond release, and exit-lock side effects are tested by DSL-030 / DSL-031 / DSL-032. The skip behaviour for `Reverted` / `Finalised` is tested by DSL-033.

## Specification

### Function Signature

```rust
impl SlashingManager {
    pub fn finalise_expired_slashes(
        &mut self,
        validator_set: &mut dyn ValidatorView,
        effective_balances: &dyn EffectiveBalanceView,
        bond_escrow: &mut dyn BondEscrow,
        reward_payout: &mut dyn RewardPayout,
        total_active_balance: u64,
    ) -> Vec<FinalisationResult>;
}
```

### Algorithm

```rust
let expired_hashes = self.book.expired_by(self.current_epoch);
let mut results = Vec::with_capacity(expired_hashes.len());
for hash in expired_hashes {
    let pending = match self.book.get_mut(&hash) {
        Some(p) => p,
        None => continue,
    };

    // Skip Reverted / Finalised — see DSL-033
    match pending.status {
        PendingSlashStatus::Reverted { .. } | PendingSlashStatus::Finalised { .. } => continue,
        _ => {}
    }

    // ... correlation penalty (DSL-030), bond release (DSL-031), exit lock (DSL-032) ...

    pending.status = PendingSlashStatus::Finalised {
        finalised_at_epoch: self.current_epoch,
    };
    results.push(FinalisationResult { /* ... */ });
}
results
```

### Transition Matrix

| Prior Status | Post Status | FinalisationResult Emitted? |
|--------------|-------------|-----------------------------|
| `Accepted` | `Finalised { current_epoch }` | ✅ |
| `ChallengeOpen { .. }` | `Finalised { current_epoch }` | ✅ |
| `Reverted { .. }` | unchanged | ❌ (see DSL-033) |
| `Finalised { .. }` | unchanged | ❌ (idempotent; see DSL-033) |

### Ordering

Per SPEC §10 epoch-boundary orchestration, `finalise_expired_slashes` runs after inactivity accounting and before `current_epoch += 1`. Within the method, the iteration order over expired hashes MUST be deterministic (ascending by `window_expires_at_epoch`, then by stable ordering within the bucket vec).

## Acceptance Criteria

- [ ] For every expired hash with `Accepted` status, post-call status equals `Finalised { finalised_at_epoch: current_epoch }`.
- [ ] For every expired hash with `ChallengeOpen` status, post-call status equals `Finalised { finalised_at_epoch: current_epoch }`.
- [ ] Exactly one `FinalisationResult` emitted per transitioned pending.
- [ ] `FinalisationResult::evidence_hash` equals the pending's hash.
- [ ] Hashes with `Reverted` / `Finalised` status NOT transitioned and NOT in the result vec.
- [ ] Iteration order is deterministic given identical book contents.
- [ ] Method is idempotent: calling it twice in the same epoch does not re-finalise or double-emit.

---

## Implementation Notes

- **Primary file:** `src/manager.rs` + `src/lifecycle.rs` (`finalise_expired_slashes`).
- **Reuses:** `PendingSlashBook::expired_by` + `::get_mut`; `ValidatorView`, `BondEscrow`, `RewardPayout` traits.
- **Edge cases:**
  - Empty book → returns empty vec.
  - `current_epoch` far beyond all windows: all pendings expire; all transition in one call.
  - Multiple hashes expiring in the same epoch: emit all results in deterministic order.
