# DSL-032: finalise_expired_slashes Schedules Exit Lock

> **Authoritative requirement:** [DSL-032](../NORMATIVE.md#DSL-032)
> **Verification:** [VERIFICATION.md](../VERIFICATION.md)
> **Tracking:** [TRACKING.yaml](../TRACKING.yaml)
> **Spec reference:** [SPEC.md](../../../../resources/SPEC.md) — Sections 2.2, 7.4 (step 4), 15.1, 22.3

---

## Summary

Once a slash is finalised, the affected validator MUST be locked out of exiting the active set for `SLASH_LOCK_EPOCHS = 100` epochs. This prevents a slashed validator from immediately withdrawing their remaining stake and escaping the tail-end of correlation penalties or future block-inclusion commitments.

`SlashingManager::finalise_expired_slashes` MUST call `ValidatorEntry::schedule_exit(current_epoch + SLASH_LOCK_EPOCHS)` on every validator in `pending.base_slash_per_validator`. The target epoch is recorded in `FinalisationResult::exit_lock_until_epoch`.

`schedule_exit` (SPEC §15.1) is the canonical API on `ValidatorEntry` for scheduling forced exits; the validator set / consensus implementation is responsible for actually enforcing the lock (refusing voluntary-exit messages until `current_epoch >= exit_lock_until_epoch`). The slashing manager only sets the target epoch — it does not validate or query exit status.

## Specification

### Call Site

```rust
// Inside finalise_expired_slashes per-pending loop, after correlation penalty:
let exit_lock_until_epoch = self.current_epoch + SLASH_LOCK_EPOCHS;

for entry in &pending.base_slash_per_validator {
    let idx = entry.validator_index;
    if let Some(v) = validator_set.get_mut(idx) {
        v.schedule_exit(exit_lock_until_epoch);
    }
}

result.exit_lock_until_epoch = exit_lock_until_epoch;
```

### Constants in scope

| Constant | Value | Source |
|----------|-------|--------|
| `SLASH_LOCK_EPOCHS` | `100` | SPEC §2.2 |

### Invariants

| Invariant | Description |
|-----------|-------------|
| Target epoch | `exit_lock_until_epoch == current_epoch + SLASH_LOCK_EPOCHS`, no off-by-one |
| Per-validator application | Every validator in `base_slash_per_validator` receives the call |
| Absolute epoch semantics | `schedule_exit` takes an absolute future epoch, not a duration |
| Result field | `FinalisationResult::exit_lock_until_epoch` equals the argument |
| Skip on Reverted/Finalised | No calls for skipped pendings (DSL-033) |

## Acceptance Criteria

- [ ] `ValidatorEntry::schedule_exit` called exactly once per validator in `pending.base_slash_per_validator`.
- [ ] Argument equals `current_epoch + SLASH_LOCK_EPOCHS`.
- [ ] `FinalisationResult::exit_lock_until_epoch` equals the scheduled value.
- [ ] For multi-validator slashes (attester), every index receives the same `exit_lock_until_epoch`.
- [ ] Skipped for `Reverted` / `Finalised` prior-status pendings (DSL-033).
- [ ] Validator missing from `validator_set` is tolerated silently (defensive skip).

---

## Implementation Notes

- **Primary file:** `src/manager.rs`.
- **Reuses:** `ValidatorEntry::schedule_exit` (SPEC §15.1).
- **Edge cases:**
  - Validator already has an earlier/later scheduled exit: `schedule_exit` contract is to set the absolute target epoch; if the consensus-layer implementation needs `max` semantics, that is its concern, not this module's.
  - `current_epoch + SLASH_LOCK_EPOCHS` overflow: for realistic `u64` epoch values this is not a concern within the lifetime of the chain.
  - Validator missing from `validator_set` (bookkeeping drift): skip silently; do not panic, do not fail the whole finalise call.
